# 小说生成质量提升方案：标点、爽点、面板、人物成长

> 适用项目：Magic Creater World（worldforger 故事生成管线）
> 目标：让 AI 生成的章节更像中文网络小说，减少标点硬伤，增强爽感，规范面板展示，并让角色成长落到数值/境界上。

---

## 一、当前现状（基于代码与资料梳理）

| 维度 | 现有能力 | 主要问题 |
|------|---------|---------|
| **标点符号** | `worldforger/punctuation_normalize.py` 已做半角/全角、引号、省略号、破折号、对话标点后处理 | 对 AI 常犯的「一逗到底」「并列成分缺顿号」「概数误加顿号」「冒号套用」「中英文标点混用」等仍无能为力；prompt 里也未把标点作为硬约束 |
| **爽点/网文节奏** | `manuscript_system` 已要求「朴素、克制、白描」、铺垫、伏笔回收、角色成长 | 偏文学化，缺少网文「爽点密度」「打脸四部曲」「起承转爽」等工业模板；容易导致章节平铺直叙 |
| **面板描述** | 角色已有 `attributes`、`inventory`、`power_tier` 等字段，生成时会被注入 prompt | 没有统一的面板输出模板，AI 可能把数值写成说明文，格式五花八门 |
| **人物成长数值化** | 生成后会更新 `runtime_state`、物理状态、知识边界、决策历史等 | **不会**自动回写基础属性 `attributes`、境界 `power_tier`、物品 `inventory`；角色变强只停留在叙事层 |

> 关键代码位置：
> - 文稿生成入口：`worldforger/story/story_service.py::generate_manuscript`
> - System prompt：`worldforger/story/story_prompts.py::manuscript_system / chapter_beats_system / polisher_system`
> - 标点后处理：`worldforger/punctuation_normalize.py`
> - 角色能力注入：`worldforger/story/story_service.py::_inject_character_capabilities`

---

## 二、方案总览

采用 **「Prompt 层约束 + 后处理修正 + 结构化抽取回写」** 三层改造：

1. **Prompt 层**：在 `manuscript_system` / `chapter_beats_system` 中新增网文爽点、面板模板、标点规范、人物数值成长指令。
2. **后处理层**：扩展 `punctuation_normalize.py` 规则，并在润色（`polisher_system`）中增加「标点/网文节奏」专项审核。
3. **数据回写层**：新增「角色成长抽取器」，在章节生成后把属性/技能/境界/物品变化写回 `characters.entities`。

下面分四个问题给出可落地的改造细节。

---

## 三、标点符号：从「程序修正」到「硬约束 + 规则兜底」

### 3.1 常见 AI 错误（资料来源：国家标准 GB/T 15834-2011 及网文编辑经验）

1. 中英文标点混用：`他说:"你好".` → `他说：“你好。”`
2. 概数加顿号：`三、五里` → `三五里`
3. 并列词语间该用顿号却用空格或逗号：`力量 敏捷 体质` → `力量、敏捷、体质`
4. 一逗到底，一个长句里全是逗号。
5. 冒号套用、引号前后不一致。
6. 省略号写成 `...` / `。。。`（后处理已部分修复）。
7. 对话标签后标点错误：`"走吧"他说` → `“走吧，”他说`

### 3.2 Prompt 层改造

在 `manuscript_system` 的「文风要求」后追加：

```text
【中文标点硬规范】
- 全文使用中文全角标点（，。！？：；""''（）——……），禁止英文半角标点夹杂在中文段落中。
- 并列词语之间用顿号：力量、敏捷、体质；并列分句之间用逗号或分号。
- 概数（三五里、七八十人）中间不加顿号。
- 对话格式统一为："……，"他说。/ "……？"她问。引号内句末点号放在引号内。
- 禁止一逗到底：每 3–4 个短句后必须使用句号或感叹号切分。
- 省略号统一为 "……"，破折号统一为 "——"，不要用 "..."、".." 或单个 "—"。
- 冒号只用于引出解释/列举/对话，一个句子内不要连续使用两个冒号。
```

### 3.3 后处理层改造

在 `worldforger/punctuation_normalize.py` 中新增规则（按优先级）：

| 规则 | 示例 |
|------|------|
| 中文内部英文标点强制全角 | `他跑.` → `他跑。` |
| 并列名词缺顿号补全 | 通过正则识别「名词 名词 名词」模式补顿号（需配合禁用词表，避免误伤） |
| 概数去顿号 | `三、五里` → `三五里` |
| 一逗到底分句 | 对超过 60 字且仅含逗号的长句，尝试在第二个谓语后插入句号（可用规则或轻量 LLM） |
| 引号内对话标签补逗号 | `"走吧"他说` → `"走吧，"他说` |

> 注：复杂的长句切分建议走 **润色阶段 LLM 专项审核**，而不是硬规则，避免误伤。

### 3.4 润色阶段强化

在 `polisher_system` 中增加一条：

```text
- 检查中文标点：修复中英文混用、顿号缺失、一逗到底、引号不匹配、对话标签标点错误。
```

## 四、爽点与网文用词：给 AI 装上「节奏模板」

### 4.1 核心原则（来自对飞卢/番茄/起点节奏的检索）

- **小爽点**：每章至少 1 个（一句打脸的话、路人震惊、解锁小奖励、小反转）。
- **中爽点**：每 3–5 章 1 个（一次完整打脸流程、阶位跃升、重要关系突破、谜题揭晓）。
- **大爽点**：每卷/每 20–30 章 1 个（Boss 战逆转、身份曝光、多伏笔回收）。
- **打脸四部曲**：挑衅 → 隐忍/准备 → 反转炸场 → 余震收尾。
- **章节结构**：起（承接+冲突）→ 承（压力升级）→ 转（转折/底牌）→ 爽（反击/收获/震惊）→ 钩子。

### 4.2 Prompt 层改造

在 `manuscript_system` 和 `chapter_beats_system` 中追加「网文爽点与节奏」块：

```text
【网文爽点与节奏要求】
- 每章至少包含 1 个"小爽点"：可以是一句反击台词、路人震惊、收获小物品、解锁提示、意外反转。
- 每 3–5 章需有 1 个"中爽点"：完整的打脸/逆袭/升级/揭晓流程。
- 爽点类型要轮换，避免连续打脸造成审美疲劳：打脸、收获、升级、解谜、情感推进、身份揭示交替使用。
- 打脸情节使用"四部曲"：反派挑衅 → 主角隐忍/准备 → 反转碾压 → 余震（围观者震惊、反派崩溃、后悔）。
- 单章结构建议"起承转爽+钩子"：
  1. 起：承接上章钩子，引出本章冲突（约 500 字）。
  2. 承：冲突升级、压力增大、反派加码（约 800 字）。
  3. 转：主角亮出底牌/发现转机/敌人露出破绽（约 300 字）。
  4. 爽：反击、收获、震惊全场 + 章末钩子（约 400 字）。
- 紧张/战斗场景多用短句（每句不超过 15 字），减少心理描写和形容词，用动作与结果推动节奏。
- 章末必须留一个钩子：悬念、反转、新信息或情绪炸弹，让读者想点开下一章。
```

### 4.3 用词/网络化改造

在 `manuscript_system` 的「文风要求」中新增：

```text
【网文化表达】
- 对话与心理活动可适当使用网络流行语感，但不过度低俗："这波血赚"、"稳了"、"离谱"、"绷不住"、"居然还能这样"。
- 收获/升级时给出明确、具体的数值或物品反馈，不要含糊地写"实力提升了一些"。
- 震惊反应要具体："瞳孔一缩"、"倒吸一口凉气"、"全场死寂"、"下巴差点掉下来"。
- 反派挑衅要够狠、够具体，让读者产生情绪代偿。
```

## 五、面板描述：统一模板与自动渲染

### 5.1 典型网文面板格式（参考系统流/修仙流作品）

常见写法：

```text
【人物面板】
姓名：林凡
境界：练气三层（12/100）
经验：87/100
属性：力量 11 | 敏捷 9 | 体质 10 | 悟性 14 | 灵根 7
状态：轻伤（右臂）
技能：基础剑诀（熟练度 23/100）、吐纳术（熟练度 41/100）
物品：劣质铁剑 ×1、回气丹 ×3
```

或更紧凑：

```text
【属性面板】
力量：11（+2）
敏捷：9
体质：10
悟性：14
技能：基础剑诀 Lv.2（23/100）
```

### 5.2 Prompt 层改造

在 `manuscript_system` 中追加：

```text
【面板/系统提示格式】
当章节中出现属性面板、系统提示、境界突破、技能升级等需要展示数据的内容时，统一使用以下格式：

【人物面板】
姓名：{角色名}
境界：{境界名}（{当前进度}/{升级阈值}）
经验：{当前经验}/{升级所需经验}
属性：{属性1} {数值} | {属性2} {数值} | ...
技能：{技能名}（熟练度 {数值}/{阈值}）、{技能名} Lv.{等级}
物品：{物品名} ×{数量}
状态：{异常状态}

【系统提示】
叮！恭喜宿主，{事件描述}。
获得：{奖励}。
{属性/技能}：{变化}。

要求：
- 数据必须和 world.json 中的 attribute_system / power_system / inventory 保持一致。
- 面板出现后，主角应基于面板信息做出下一步行动或心理反应，不要只展示数据就结束。
- 数值变化使用 "+X" / "-X" 标出本次变化。
```

### 5.3 代码层建议

- 在 `worldforger/story/story_prompts.py` 中新增 `panel_template()` 函数，返回上述模板字符串，便于多处复用。
- 在 `_inject_character_capabilities` 中把当前属性值以「面板」形式再输出一次，让 AI 更容易照搬。
- 可在前端为 `manuscript` 渲染增加面板样式（等宽字体、边框、暗色适配），与正文区分。

## 六、人物成长数值化：从「叙事成长」到「数据成长」

### 6.1 目标

让角色在情节推进后，**基础属性、境界、技能熟练度、物品** 真正发生变化，并在后续章节中被 prompt 重新注入。

### 6.2 新增：角色成长抽取器

在 `worldforger/story/story_prompts.py` 中新增 prompt：

```python
def character_progression_extract_system() -> str:
    return (
        "你是角色成长分析助手，负责从本章正文中提取所有出场角色的实力/属性/技能/境界/物品变化。\n"
        "只输出 JSON，不要输出其他文字。\n"
        "JSON 格式：\n"
        "{\n"
        '  "updates": [\n'
        '    {\n'
        '      "char_id": "角色id",\n'
        '      "name": "角色名",\n'
        '      "attributes_delta": {"stat_id": +N, ...},\n'
        '      "skill_changes": [{"skill_name": "...", "delta": +N, "new_level": "..."}],\n'
        '      "tier_change": {"from": "...", "to": "..."} 或 null,\n'
        '      "inventory_changes": [{"name": "...", "action": "gain|lose|use", "quantity": N}],\n'
        '      "growth_text": "用一句话概括本章成长（用于摘要）"\n'
        '    }\n'
        '  ]\n'
        "}\n"
        "注意：\n"
        "- 仅列出本章中实际发生变化的角色；未出场或无变化的角色不要列出。\n"
        "- 数值变化必须基于正文明确描写，不要编造。\n"
        "- 如果正文没有具体数值，请根据情节合理推断小幅变化（±1~3）。\n"
        "- tier_change 只有在正文明确突破境界时才填写。\n"
    )
```

### 6.3 回写逻辑

在 `worldforger/story/story_service.py` 的章节生成后处理中新增：

```python
async def _try_update_character_progression(world, chapter_id, manuscript_text):
    """Extract character progression from manuscript and write back to world.json."""
    # 1. 调用 LLM 获取 progression JSON
    # 2. 对 updates 中每个 char_id：
    #    - attributes_delta 加到 characters.entities[].attributes
    #    - skill_changes 更新到 notable_skills 或 tier skill_tree 熟练度
    #    - tier_change 更新 power_tier
    #    - inventory_changes 更新 inventory[]
    #    - 在 runtime_state 中记录 growth_text
    # 3. 保存 world.json
```

### 6.4 与前文能力注入联动

`_inject_character_capabilities` 已经会把 `attributes` 注入 prompt。只要回写成功，下一章生成时角色就是新的数值，**成长会自然延续**。

### 6.5 升级节奏建议

为避免数值膨胀，prompt 中可约定：

```text
【升级节奏】
- 小事件：属性 ±1~2，技能熟练度 +3~5。
- 中事件（Boss 战/重要副本）：属性 ±3~5，可能解锁新技能。
- 大事件（卷高潮）：境界突破或获得核心物品。
- 不要让主角在单章内从凡人到无敌；成长要分阶段、有代价。
```

## 七、实施步骤建议（按优先级）

### 第一阶段：立竿见影（1–2 天）

1. 修改 `manuscript_system`：追加「中文标点硬规范」「网文爽点与节奏」「面板/系统提示格式」「网文化表达」四个 prompt 块。
2. 修改 `chapter_beats_system`：追加爽点/钩子/起承转爽结构要求，让细纲先具备节奏。
3. 修改 `polisher_system`：追加标点和爽点节奏检查。
4. 扩展 `punctuation_normalize.py`：补全顿号、概数、中英文标点、对话标签规则。

### 第二阶段：数据闭环（3–5 天）

5. 新增 `character_progression_extract_system` prompt 与 `_try_update_character_progression` 回写函数。
6. 在 `generate_manuscript` 的后处理链中调用该函数（可默认开启，也可由 `StoryWritingDefaults` 开关控制）。
7. 在前端「角色」卡片中展示 `attributes_delta` / `growth_text`，让用户可见成长轨迹。

### 第三阶段：模板与样式（可选）

8. 在 `worldforger/story/story_prompts.py` 中把面板模板独立为 `panel_template()` 函数。
9. 前端增加面板渲染样式（等宽字体、边框、暗色模式适配）。
10. 根据用户反馈迭代 prompt 块权重，必要时训练/微调小模型做标点审核。

## 八、风险与注意事项

- **不要破坏非爽文模式**：上述 prompt 块建议只在 `creative_mode == "novel"` 或用户开启「网文模式」时注入；`coc` / `dnd` 等跑团模式保持克制。
- **数值膨胀控制**：成长抽取器必须有 clamp 和上限校验，参考 `attribute_system.stats[].reference_percent` 与 `power_system.tiers` 的层级上限。
- **面板不要喧宾夺主**：每章面板出现 1–2 次即可，过多会打断叙事。
- **标点后处理要保守**：一逗到底/长句切分建议走 LLM 润色，规则引擎只处理确定性错误，避免过度改写作者原创内容。

## 九、附录：可直接试用的 prompt 块

### A. 网文爽点与节奏块

```text
【网文爽点与节奏要求】
- 每章至少 1 个小爽点；每 3–5 章 1 个中爽点；每卷 1 个大爽点。
- 爽点类型轮换：打脸、收获、升级、解谜、情感推进、身份揭示。
- 打脸四部曲：挑衅 → 隐忍/准备 → 反转碾压 → 余震（围观者震惊、反派崩溃）。
- 单章结构建议「起承转爽+钩子」：起 500 字、承 800 字、转 300 字、爽 400 字、钩子 1–2 句。
- 战斗/紧张场景多用短句，每句不超过 15 字。
- 章末必须留钩子。
```

### B. 面板模板块

```text
【面板/系统提示格式】
当需要展示数据时，使用以下格式：

【人物面板】
姓名：{角色名}
境界：{境界名}（{进度}/{阈值}）
属性：{属性名} {数值} | {属性名} {数值}
技能：{技能名}（熟练度 {数值}/{阈值}）
物品：{物品名} ×{数量}
状态：{状态}

数值变化用 +N / -N 标注。
```

### C. 标点规范块

```text
【中文标点硬规范】
- 全文使用中文全角标点；并列词语间用顿号；概数不加顿号。
- 对话格式："……，"他说。引号内句末点号在引号内。
- 禁止一逗到底，每 3–4 短句后用句号或感叹号切分。
- 省略号用"……"，破折号用"——"，不用英文省略号或单破折号。
```
